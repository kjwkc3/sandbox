LLM Animation Validation Plan

The goal is to provide an LLM with a reliable, video-like artifact for validating animation and input-driven behavior, rather than continuing to debug a custom GIF encoder.

Current State

  • The renderer already captures frame images in debug/frames/frame_*.png.
  • There is a custom GIF writer in gif/gif.odin, but it is unstable.
  • The useful output is the captured frame sequence, not the custom encoder output.

Recommended Approach

  1. Keep the frame capture pipeline in the application.
  2. Stop using gif/gif.odin as the primary output mechanism for LLM validation.
  3. Convert the captured frames into an animation using external tools.

Why This Is Better

  • It directly solves the actual goal: producing an animation artifact.
  • It avoids low-level GIF codec debugging.
  • It leverages existing, reliable tools such as ffmpeg or ImageMagick.
  • It is easier to maintain and more portable.

Proposed Workflow

  1. Capture frames in the app and write them to debug/frames.
  2. Add external conversion commands to create a final animation file:
    • GIF: convert -delay 5 -loop 0 debug/frames/frame_*.png debug/animation.gif
    • MP4: ffmpeg -framerate 12 -i debug/frames/frame_%03d.png -c:v libx264 -pix_fmt yuv420p debug/animation.mp4
  3. Expose the workflow with build targets such as make debug, make frames, and make video.
  4. Optionally add metadata logging for input events and frame timing if the LLM needs to validate input sequences.

Handoff Instructions

  1. Validate and improve the frame capture/export pipeline.
  2. Implement build targets and conversion commands.
  3. Create a stable animation artifact from the captured frames.
  4. Optionally add metadata output for input/action validation.

Summary

Focus on reliable frame capture and external animation generation. Avoid spending more time on the custom GIF encoder until the animation workflow is stable.
